# KAN_TQNet 逐步构建式消融设计

## 1. 设计目标

这组实验不再回答“去掉某个模块会怎样”，而是回答“模型性能是如何一步步建立起来的”。
因此建议把 `KAN_TQNet` 拆成从简到繁的构建链条，逐步加入核心模块。

相比单模块删除式消融，这种设计更适合当前模型，原因是：

- 模块之间存在明显耦合，尤其是 `HDA`、`TimeMixer`、`频域分支`。
- 现有单次结果中，个别模块删除后会出现轻微波动，逐步构建更容易展示总体增益趋势。
- 论文叙事更自然，可以直接写成“从基础骨干到完整模型”的性能提升路径。

## 2. 推荐实验链条

建议按下面 9 个阶段做逐步构建：

| 阶段 | 标记 | 模型配置 | 设计含义 |
| --- | --- | --- | --- |
| 0 | `P0_backbone_mlp` | 关闭 `KAN/TQ/通道头/通道适配器/频域/趋势/HDA/电负荷细化/TimeMixer` | 最小骨干基线 |
| 1 | `P1_plus_kan` | 在 P0 基础上加入 `KAN` | 验证 KAN 非线性骨干的贡献 |
| 2 | `P2_plus_channel` | 在 P1 基础上加入 `通道头 + 通道适配器` | 验证多任务通道解耦建模 |
| 3 | `P3_plus_tq` | 在 P2 基础上加入 `TQ` | 验证 Temporal Query 机制 |
| 4 | `P4_plus_freq` | 在 P3 基础上加入 `频域分支` | 验证频域建模收益 |
| 5 | `P5_plus_trend` | 在 P4 基础上加入 `趋势残差` | 验证趋势补偿作用 |
| 6 | `P6_plus_hda` | 在 P5 基础上加入 `HDA` | 验证层次化动态聚合 |
| 7 | `P7_plus_ele_refine` | 在 P6 基础上加入 `电负荷细化` | 验证目标通道专属细化 |
| 8 | `P8_plus_timemixer` | 在 P7 基础上加入 `TimeMixer` | 完整模型终态 |

## 3. 为什么按这个顺序

推荐顺序遵循“骨干表征 -> 输入/通道增强 -> 辅助分支 -> 目标细化”的逻辑：

- `KAN` 是主干表示能力的核心，应最先加入。
- `通道头 + 通道适配器` 属于输出侧的多变量建模增强，适合作为第二层能力。
- `TQ` 是时序相关性建模模块，应放在主干能力之后。
- `频域分支` 与 `趋势残差` 都属于辅助时序分解建模，因此放在中段。
- `HDA` 是更强的高低层级路由机制，建议在基础时序建模能力建立后再加入。
- `电负荷细化` 和 `TimeMixer` 更偏向后处理与精修，适合作为后段增益。

## 4. 运行设置建议

### 固定设置

- 预测步长统一使用：`24, 48, 72, 96`
- 训练轮数统一使用：`60`
- 数据划分、优化器、学习率、损失权重保持一致
- 每一组实验仅修改对应模块开关，其余设置严格不变

### 强烈建议

- 每个阶段至少运行 `3` 个随机种子
- 推荐种子：`2020, 2021, 2022`
- 论文中报告 `mean ± std`

原因：

- 你当前已有消融中出现了“删掉某模块反而略优”的情况
- 这通常说明单次运行波动不可忽略
- 多种子平均后，结论会更稳，也更适合论文答辩

## 5. 论文结果表建议

### 表A：逐步构建主表

行设置为 `P0 ~ P8`，列设置为 `24 / 48 / 72 / 96`，单元格建议填：

- `MAE`
- 或 `MAE / RMSE`
- 若版面允许，可写成 `MAE (Δ)`，其中 `Δ` 是相对上一阶段或相对 `P0` 的变化

### 表B：增益表

专门统计每一步相对上一阶段的增益：

- `P1 - P0`
- `P2 - P1`
- `...`
- `P8 - P7`

这样可以更直观展示：

- 哪一步是主要增益来源
- 哪一步在长预测步长上收益最大

## 6. 预期论文叙事

建议正文按下面逻辑写：

1. 从最小骨干 `P0` 出发，建立基础预测能力。
2. 加入 `KAN` 后，模型非线性表达能力增强，性能显著提升。
3. 加入 `通道头/通道适配器` 与 `TQ` 后，多变量和时序相关性建模进一步增强。
4. 加入 `频域分支`、`趋势残差` 后，中长时预测能力进一步改善。
5. 加入 `HDA` 与 `TimeMixer` 后，模型在 `72/96` 步长下获得最明显收益，说明层次化动态聚合和低频精修是长时预测性能提升的关键。

## 8. 当前设计的边界

这套逐步构建主要针对“当前默认开启、且确实构成完整模型核心增益链”的模块。

不建议放进主链条的模块：

- `use_patch_multiscale`
- `use_multi_scale`
- `use_local_conv`

原因：

- 你当前默认配置里这些模块并未构成稳定主路径
- 现有结果中它们要么未启用，要么对指标影响极小
- 更适合作为补充性附录消融，而不是主表核心链条
